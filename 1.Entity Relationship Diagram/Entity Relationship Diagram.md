# Credit Card Approval Prediction - Database ERD Design & Implementation Report

This combined report serves as both the **Implementation Plan** and the final **Walkthrough Report** for the Credit Card Approval Prediction System database design task.

---

## 1. Project Objective & Scope

The objective of this task is to design an Entity Relationship Diagram (ERD) representing the database structure of the Credit Card Approval Prediction System. The system manages:
- System users (administrators/analysts) who access and manage the system.
- Applicant personal and financial profile details.
- Previous credit history payment records.
- Metadata and files for trained machine learning models.
- Prediction outputs generated by the machine learning models.

---

## 2. Database Schema Details

### Main Entities & Attributes

#### 1. `Users`
Stores authentication and roles for users who access the system.
- **`UserID`** (Primary Key, `INT`)
- **`Name`** (`VARCHAR(100)`)
- **`Email`** (`VARCHAR(100)`)
- **`Password`** (`VARCHAR(255)`)
- **`Role`** (`VARCHAR(50)`)

#### 2. `Applicant_Details`
Stores demographic, financial, and employment information submitted by applicants.
- **`ApplicantID`** (Primary Key, `INT`)
- **`UserID`** (Foreign Key, `INT` -> `Users.UserID`)
- **`IncomeType`** (`VARCHAR(50)`)
- **`EducationType`** (`VARCHAR(100)`)
- **`FamilyStatus`** (`VARCHAR(50)`)
- **`HousingType`** (`VARCHAR(50)`)
- **`EmploymentDays`** (`INT`)

#### 3. `Credit_History`
Maintains applicants' previous credit history records, used for training models.
- **`HistoryID`** (Primary Key, `INT`)
- **`ApplicantID`** (Foreign Key, `INT` -> `Applicant_Details.ApplicantID`)
- **`MonthsBalance`** (`INT`)
- **`PaymentStatus`** (`VARCHAR(20)`)
- **`OverdueStatus`** (`VARCHAR(50)`)

#### 4. `ML_Model`
Stores details of the machine learning models trained and used for evaluation.
- **`ModelID`** (Primary Key, `INT`)
- **`ModelName`** (`VARCHAR(100)`)
- **`AlgorithmType`** (`VARCHAR(100)`)
- **`Accuracy`** (`FLOAT`)
- **`ModelFile`** (`VARCHAR(255)`)

#### 5. `Approval_Prediction`
Stores prediction results generated by the machine learning models.
- **`PredictionID`** (Primary Key, `INT`)
- **`ApplicantID`** (Foreign Key, `INT` -> `Applicant_Details.ApplicantID`)
- **`ModelID`** (Foreign Key, `INT` -> `ML_Model.ModelID`)
- **`ApprovalResult`** (`VARCHAR(20)`)
- **`RiskCategory`** (`VARCHAR(50)`)
- **`PredictionDate`** (`TIMESTAMP`)

---

## 3. Relationships & Cardinality

| Parent Entity | Child Entity | Relationship Type | Cardinality | Explanation |
| :--- | :--- | :--- | :--- | :--- |
| **`Users`** | **`Applicant_Details`** | One-to-Many | `1 ───< N` | A user (analyst) can manage multiple applicant submissions. |
| **`Applicant_Details`** | **`Credit_History`** | One-to-Many | `1 ───< N` | An applicant can have multiple historical payment balance records. |
| **`Applicant_Details`** | **`Approval_Prediction`** | One-to-One | `1 ─── 1` | Each applicant profile evaluation maps to exactly one prediction record. |
| **`ML_Model`** | **`Approval_Prediction`** | One-to-Many | `1 ───< N` | One machine learning model can evaluate and generate multiple predictions. |
